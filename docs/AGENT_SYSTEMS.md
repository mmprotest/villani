# Agent-system and harness contract

PT6 models a coding route as a complete agent system rather than a model name. The canonical
identity is `villani.agent_system.v1`; it binds the non-secret harness, adapter, protocol, model,
provider endpoint, serving metadata, execution environment, permissions, repository/task profile,
verification policy, capabilities, qualification references, and billing knowledge into a
content-addressed `asys_<sha256>` ID.

Secrets are removed before the configuration digest is calculated. Endpoint identity excludes
credentials, query strings, and fragments. Unknown revisions, costs, environment fingerprints,
and capability facts remain explicit unknowns; they are never replaced by fabricated values.

## Production boundary

Villani Code, Codex, and Claude Code have production adapters. Codex uses the stable app-server
JSON-RPC stdio lifecycle and validates both directions against schemas generated by the exact
installed CLI. Claude Code uses its supported non-interactive `stream-json` lifecycle with a
controlled prompt file, strict settings, no session persistence, and bounded process supervision.
Neither adapter scrapes terminal presentation, and neither uses Codex's experimental WebSocket
transport.

PT6 supports Codex CLI `>=0.144.0,<0.145.0` and Claude Code `>=2.1.138,<2.2.0`. Exact versions are
probed again at execution. A version outside the range, an installed schema mismatch, missing
authentication, or an unavailable isolation boundary disables the route with one repair action.
Claude Code's own command sandbox is unavailable on native Windows; use WSL2 or a configured outer
container/devcontainer. Villani never weakens that boundary silently.

New Codex and Claude Code identities remain `experimental` or `provisional`. Executing a task or
passing protocol conformance does not mark either system `qualified` and does not claim Gate C.
Custom provider and local-model substitution are unsupported. A custom model is exposed only after
an exact version/provider/model/protocol conformance record; otherwise configure the special
`default` model so the vendor chooses and reports its actual model.

Legacy `backends` configuration remains readable and is preserved. On load or an atomic
configuration write, each coding backend receives a generated Villani Code agent-system entry.
Generated entries are removed only when their corresponding legacy backend is removed; explicitly
authored entries are preserved. The migration report records these decisions.

Use the public inventory commands without running a coding task:

```text
villani agents list
villani agents inspect <route-or-system-id>
villani agents doctor [route-or-system-id]
```

Each command also supports `--json`. Doctor exits nonzero for a disabled, unsupported, unqualified,
or unhealthy route.

`agents list` also reports unconfigured Villani Code, Codex, and Claude Code discoveries: installed
state, exact version, authentication readiness, protocol, model/provider capability, conformance,
qualification state, and the next repair action. It never asks for an internal capability score.

## PT6 route configuration

Vendor authentication remains in vendor-owned configuration. Do not copy tokens into Villani
configuration. A minimal Codex route uses `provider: openai`; Claude Code uses
`provider: anthropic`:

```yaml
backends:
  codex-default:
    provider: openai
    model: default
    roles: [coding]
    command_name: codex
    billing_mode: unknown
  claude-default:
    provider: anthropic
    model: default
    roles: [coding]
    command_name: claude
    billing_mode: unknown

agent_systems:
  schema_version: villani.agent_system_configuration.v1
  systems:
    codex-default:
      backend: codex-default
      production_enabled: true
      harness:
        id: codex
    claude-default:
      backend: claude-default
      production_enabled: true
      harness:
        id: claude-code
```

On native Windows the Claude route remains disabled until its execution environment is a container
or devcontainer (or an exact strict-sandbox conformance record is supplied). For a non-default
model, `model_conformance` must bind `status: passed`, exact harness version, provider, model,
protocol, and a `sha256:` report digest. This is identity evidence, not production qualification.

## Adapter lifecycle

`villani.harness_adapter.v1` defines these operations:

```text
probe -> describe capabilities -> prepare session -> execute task
      -> stream normalized events -> collect result and artifacts
      -> request cancellation when needed -> cleanup
doctor (out-of-band diagnostics)
```

The contract fixes protocol negotiation, session IDs, cancellation and timeout behavior, message
bounds, backpressure policy, permission requests, separated stdout/stderr, event ordering, failure
classification, artifacts, and cleanup. Unknown namespaced raw events are retained alongside the
normalized event stream.

Capabilities are tri-state (`supported`, `unsupported`, or `unknown`) and carry declared,
detected, or conformance evidence. In particular, a generic model selector does not imply custom
model or custom provider support.

## Harness-neutral evidence

Every attempt emits `villani.harness_result.v1`, including:

- exact isolated worktree and baseline digest;
- patch and normalized changed paths;
- separated stdout and stderr;
- ordered normalized events and a raw-trace reference;
- exact acknowledged harness version, protocol/schema, session/thread, provider, model, and effort;
- usage, cost, and duration with accounting status;
- harness and infrastructure status;
- artifacts and cleanup result.

The verifier receives task, requirements, patch, changed files, validation evidence, and execution
environment facts. It does not receive harness identity, route, cost, or competing-candidate facts.
Only acceptance-eligible verification reaches selection.

## Run artifacts and compatibility

New bundles contain:

```text
agent-systems/index.json
agent-systems/<system-id>.json
agent-systems/migration.json
attempts/<attempt-id>/harness-result.json
```

The manifest lists agent-system IDs and the attempt snapshot links the selected identity and
harness result. All new fields are optional in the version-1 run schemas, so founder-era bundles
remain readable without rewriting or guessing missing identity evidence.

`villani.harness_conformance_report.v1` records the fail-closed conformance kit. It covers
manifest/protocol negotiation, exact versions, successful and empty patches, recovery, worktree and
path safety, ordering, permissions, cancellation, timeout, rate limits, malformed/oversized output,
schema drift, missing results, partial patches on crash, authoritative and unknown cost, non-ASCII
and spaced paths, cleanup, and secret redaction. All checks passing is necessary evidence; PT6 does
not treat it as sufficient to qualify Codex or Claude Code.

## ACP

The reusable ACP v1 client implements newline-delimited JSON-RPC stdio initialization,
capabilities, sessions, updates/diffs, worktree-confined file operations, bounded terminals,
permissions, `session/cancel`, JSON-RPC cancellation, backpressure, process supervision, and
redacted raw events. No production route selects ACP merely because it connects. An ACP adapter
must separately prove exact harness/model identity and full trace/evidence parity.

## PT7 repository qualification

PT7 replaces the provisional global label as a routing decision with a derived repository/task
assessment. The immutable qualification ledger binds every observation to repository lineage,
task category/difficulty/risk/capabilities, the complete agent-system and execution identity,
verification-policy version, exact software versions, and time. Installed or connected systems do
not qualify themselves.

Automatic routing can select only `qualified` systems. If no qualified route exists, it selects at
most the strongest otherwise-eligible `provisional` fallback and records the fallback. An
`experimental` route needs `--agent-system ... --allow-experimental`; the displayed and persisted
override cannot create qualification. `unsupported` is never overridable.

Use `villani agents qualify`, `status`, `evidence`, `invalidate`, and `gate-c` to manage and inspect
evidence. The Agents page shows the current repository-specific state, exact harness/model,
observed acceptance, known median accepted-change cost, duration, sample, last tested, caveat,
Doctor action, and evidence action. Unknown values remain visibly unknown rather than becoming
zero or a global score.

See [Repository-specific qualification](REPOSITORY_QUALIFICATION.md) for evidence eligibility,
conservative backoff, default thresholds, drift, migration, and Gate C.

## PT8 accepted-change routing

PT8 leaves repository qualification as the fail-closed eligibility boundary, then orders eligible
routes by conservative total cost per accepted change. The calculation includes execution,
verification, human review, retry/escalation, and an optional latency penalty divided by the PT7
Wilson lower bound. Missing inputs remain explicit and trigger a strongest-evidence fallback rather
than becoming zero.

Each controller decision now has a versioned route-plan artifact with alternatives, rejections,
unknowns, reserves, the selected sequence, deterministic policy identity, and the user-facing
explanation. Finalized eligible outcomes update future qualification/economics profiles; forced,
infrastructure-excluded, corrupt, secret-affected, or unverified outcomes do not train automatic
policy. Policy publication requires point-in-time replay and supports pointer-based rollback.

See [Accepted-change economics routing](ACCEPTED_CHANGE_ROUTING.md) for the objective, constraints,
CLI lifecycle, online evidence policy, compatibility behavior, and scorecards.
